# MX10

This document records the implementation decisions and calibration history for
the MX10 / GT01-like BLE printer.

## Current implementation

- Device module: `src/bt_printers/devices/mx10.py`
- Protocol module: `src/bt_printers/devices/mx10_protocol.py`
- Registry aliases: `mx10`, `cat`
- Transport: Bluetooth LE through `bleak`
- Send path: one encoded print job, split by BLE MTU in `send_print_job()`

## BLE profile

- Advertised services:
  - `0000ae30-0000-1000-8000-00805f9b34fb`
  - `0000af30-0000-1000-8000-00805f9b34fb`
- Write characteristic:
  - `0000ae01-0000-1000-8000-00805f9b34fb`
- Notify characteristic:
  - `0000ae02-0000-1000-8000-00805f9b34fb`
- Ready notification:
  - `51 78 ae 01 01 00 00 00 ff`
- Effective print width:
  - `384 px`

## Prepare

The MX10 uses the shared preparation helpers:

- `rasterize_text()` for text-to-bitmap rendering.
- `resize_image_to_width()` for image scaling.
- `image_to_rows()` for threshold or Floyd-Steinberg raster conversion.

MX10 does not currently enable automatic average-density limiting by default.
The tested output quality was acceptable with the shared raster settings.

## Calibrate

Current calibration in `MX10_CALIBRATION`:

- `paper_width_mm=55.0`
- `left_margin_mm=3.0`
- `right_margin_mm=3.0`
- `top_margin_mm=0.0`
- `bottom_margin_mm=4.0`
- `dots_per_mm_x=8.0`
- `dots_per_mm_y=8.0`
- `feed_lines=0`
- `set_paper_repeats=1`

The calibrated image width is `384 px`. Attempts to use wider rows, including
`392 px`, caused cropped or local-region image output. The stable choice is to
stay at the printer profile width.

## Print protocol

The MX10 job builder emits this command sequence:

1. Get device state.
2. Set quality to 200 DPI.
3. Set energy.
4. Apply energy.
5. Send lattice start.
6. Send each raster row.
7. Send feed-paper command.
8. Repeat `CMD_SET_PAPER`.
9. Send lattice end.
10. Get device state.

Rows are encoded per row. The protocol first tries run-length encoding and
falls back to byte encoding when byte encoding is shorter.

## Calibration observations

Bottom-margin experiments:

| `feed_lines` | `set_paper_repeats` | Observed bottom |
| ---: | ---: | ---: |
| 0 | 3 | +17 mm |
| 0 | 0 | -8 mm |
| 128 | 0 | -8 mm |
| 0 | 2 | +16.5 mm |
| 0 | 1 | +4 mm |
| 32 | 1 | +4 mm |

Inferences:

- `cmd_feed_paper()` did not affect the end-of-image state in these tests.
- `CMD_SET_PAPER` is not a linear paper-advance command.
- One `CMD_SET_PAPER` repeat plus `4 mm` of raster blank rows gave the stable
  final result.

Observed calibrated output:

- Top margin: about `8 mm`
- Bottom margin: about `8 mm`
- Left margin: about `3 mm`
- Right margin: about `5 mm`
- Image output: complete, not cropped

## Known caveats

- Do not increase the MX10 image width beyond `384 px` without re-testing on
  paper.
- `feed_lines` should not be treated as reliable trailing feed for this device.
- The right margin is slightly larger than the configured left/right margins,
  probably due to the printer's physical printhead or paper path alignment.
